Detailed Description

Physical Interface Layer to the Module using the Universal Asynchronous Receiver/Transmitter interface. This file acts as an interface between the module library methods and the hardware (hal file).

Note:
To output SPI transmit information, define ZM_PHY_UART_VERBOSE. It is recommended to define this on a per-project basis. In IAR, this can be done in Project Options : C/C++ compiler : Preprocessor In the defined symbols box, add: ZM_PHY_UART_VERBOSE

Troubleshooting

Module UART requires 115,200 kbs, with flow control enabled. Most problems with the UART interface are caused by flow control errors, so check these first.

Concurrency

To prevent code in two places accessing the messageBuffer at the same time, we use a mutex. This mutex is implemented as a bit flag in the messageFlags variable. Accessing the mutex is through a spinlock, which waits until the buffer becomes available before using it. For example: 

  while (messageFlags & MESSAGE_BUFFER_IN_USE);   //wait for msgBuffer to be available
  messageFlags |= MESSAGE_BUFFER_IN_USE;          //lock the buffer

enum messageBufferStates

The states of the messageBuffer receive state machine. This is private and should not be needed to be accessed outside this file

Enumerator:
NOT_STARTED 

awaiting the start of a new message

STARTED 

a message has started, but not yet completed

AWAITING_FCS 

all the bytes of message body received, need to check FCS

COMPLETE 

FCS check complete, message can now be read

ERROR 

there was an error in parsing

Function Documentation

void auxSerialPortHandler ( char  c)

Processes bytes received from the Module. Should be called in the serial port ISR for the module serial port. For example: 

#pragma vector=USCIAB1RX_VECTOR
__interrupt void USCI1RX_ISR(void)
{
  if (UC1IFG & UCA1RXIFG)
  {
    auxSerialPort(UCA1RXBUF);    //reading this register clears the interrupt flag                  
  }
}

Message processing is handled according to a simple state machine. If a message has already started, then append bytes to messageBuffer. When a byte is received:

  1. Assert RTS so that the module doesn't send another byte while we're processing this one
  2. Set messageFlags.MESSAGE_BUFFER_IN_USE. This is a simple mutex to control access to messageBufferIndex, messageBuffer, and fcs to prevent concurrency issues
  3. Process received byte based on the state of messageBufferState
  4. Clear messageFlags.MESSAGE_BUFFER_IN_USE mutex
  5. Deassert RTS to allow more bytes to be received
    Parameters:
    cthe byte that was received in the ISR.
void displayMessageBuffer ( )

Displays the contents of the messageBuffer. Locks the messageBuffer while reading to prevent a concurrency problem.

Note:
If you have sufficient memory and the printf is slow then it would be better to copy messageBuffer to a new buffer and then printf that. Or use a circular buffer on the debug UART.

void resetMessage ( )

Clears the messageFlags and sets messageBufferIndex back to zero. After this method is called we are ready to receive a new message from the Module.

moduleResult_t sendMessage ( )

Public method to send messages to the Module. This will send one message and then receive the Synchronous Response (SRSP) message from the Module to indicate the command was received.

Precondition:
zmBuf contains a properly formatted message
Module has been initialized
Postcondition:
buffer zmBuf contains the response (if any) from the Module.
moduleResult_t sendSreq ( )

Sends a Module Synchronous Request (SREQ) message and retrieves the response. A SREQ is a message to the Module that is immediately followed by a Synchronous Response (SRSP) message from the Module. As opposed to an Asynchronous Request (AREQ) message, which does not have a SRSP. This is a private method that gets wrapped by sendMessage().

Precondition:
Module has been initialized
zmBuf contains a properly formatted message. No validation is done.
Postcondition:
received data is written to zmBuf
