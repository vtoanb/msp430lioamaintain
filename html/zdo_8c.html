Detailed Description

Methods that implement the Zigbee Device Objects (ZDO) interface.

The AF/ZDO interface is a more powerful version of the Simple API and allows you to configure, send, and receive Zigbee data. This file acts as an interface between the user's application and the Module physical interface. Module interface could be either SPI or UART. Refer to Interface Specification for more information.

Note:
For more information, define ZDO_VERBOSE. It is recommended to define this on a per-project basis. In IAR, this can be done in Project Options : C/C++ compiler : Preprocessor In the defined symbols box, add: ZDO_VERBOSE

moduleResult_t zdoManagementNetworkDiscoveryRequest ( uint16_t  destinationAddress,
uint32_t  channelMask,
uint8_t  scanDuration,
uint8_t  startIndex 
) 
Request that the destination device performs an active network scan by sending out beacons.

Unlike zdoNetworkDiscoveryRequest, this function MAY be used while the device is ON the network. Note that if the device is doing a scan on other RF channels then it will temporarily not be able to receive messages since the radio may not be on the same channel. In this case, another device trying to send a message to this device while it's busy scanning will return an error of 0x19.

Parameters:
destinationAddressthe short address of the device which we want to do the scan.
channelMaskbitmap of which channels to scan. If channelMask = ANY_CHANNEL then the radio will scan all channels. Must be between MIN_CHANNEL_MASK and MAX_CHANNEL_MASK, or ANY_CHANNEL.
scanDurationhow long to scan each channel - must be a BEACON_ORDER_x value. Refer to zdo.h for a list of allowed values. Available values are BEACON_ORDER_15_MSEC to BEACON_ORDER_4_MINUTES. Usually we use BEACON_ORDER_480_MSEC, but you may want to adjust for your application.
startIndexwhere to start in the response array list. The result can contain more entries than can be reported, so this field allows the user to retrieve the responses anywhere in the array list.
Postcondition:
A ZDO_MGMT_NWK_DISC_RSP message will be received containing the results of the scan
Note:
example usage to have the coordinator do a scan: zdoManagementNetworkDiscoveryRequest(0x0000, ANY_CHANNEL_MASK, BEACON_ORDER_480_MSEC, 0)
The time that it takes to run a scan is equal to (scan time) * (number of channels). So if using ANY_CHANNEL and BEACON_ORDER_480_MSEC then it will take approximately 7 seconds.
Returns:
the standard error code

moduleResult_t zdoNetworkDiscoveryRequest ( uint32_t  channelMask,
uint8_t  scanDuration 
) 
Performs an active network scan by sending out beacons. The device must NOT be on a network. If

it is then an error will be returned.

Parameters:
channelMaskbitmap of which channels to scan. If channelMask = ANY_CHANNEL then the radio will scan all channels. Must be between MIN_CHANNEL_MASK and MAX_CHANNEL_MASK, or ANY_CHANNEL.
scanDurationhow long to scan each channel - must be a BEACON_ORDER_x value. Refer to zdo.h for a list of allowed values. Available values are BEACON_ORDER_15_MSEC to BEACON_ORDER_4_MINUTES. Usually we use BEACON_ORDER_480_MSEC, but you may want to adjust for your application.
Postcondition:
A ZDO_BEACON_NOTIFY_IND message will be received for each network found, followed by a ZDO_NWK_DISCOVERY_CNF when the scan is complete.
Note:
example usage: zdoNetworkDiscoveryRequest(ANY_CHANNEL_MASK, BEACON_ORDER_480_MSEC)
The time that it takes to run a scan is equal to (scan time) * (number of channels). So if using ANY_CHANNEL and BEACON_ORDER_480_MSEC then it will take approximately 7 seconds.
Returns:
the standard error code, including:
  • MODULE_SUCCESS if success
  • 0x02:Invalid Parameter
  • 0xC2:ZNwkInvalidRequest if the device is already on the network then use ZDO_MGMT_NWK_DISC_REQ instead, or just leave the network with a ZDO_MGMT_LEAVE_REQ command.
  • 0xFC: MAC scan in progress
  • 0x1A: no memory resource available inside the module

moduleResult_t zdoStartApplication ( ) 
Starts the Zigbee stack in the Module using the settings from a previous afRegisterApplication().

After this start request process completes, the device is ready to send, receive, and route network traffic.

Note:
On a coordinator in a trivial test setup, it takes approximately 300mSec between sending START_REQUEST and receiving START_REQUEST_SRSP and then another 200-1000mSec from when we receive START_REQUEST_SRSP to when we receive START_CONFIRM. Set START_CONFIRM_TIMEOUT based on size of your network.
ZDO_STARTUP_FROM_APP field StartDelay not used
Precondition:
afRegisterApplication() was a success.
Postcondition:
We will see Device Status change to DEV_ROUTER, DEV_ZB_COORD, or DEV_END_DEVICE correspondingly if everything was ok.
